# Testing Conventions

FP+ES (関数型プログラミング + イベントソーシング) アーキテクチャにおけるテスト戦略とコーディング規約です。
テストを **UnitTest**、**IntegrationTest**、**ComponentTest**
の3つに分類し、テストピラミッドの順（スコープが狭く高速なものから、スコープが広く重いものへ）でそれぞれの責務と実装スタイルを定義します。

---

## 1. テストの分類と責務

| テスト種別               | 対象 (SUT)                                                                    | テストの目的                                    | 推奨スタイル (Kotest)                         | 状態・プロセス外連携の扱い                                       |
|:--------------------|:----------------------------------------------------------------------------|:------------------------------------------|:----------------------------------------|:----------------------------------------------------|
| **UnitTest**        | Adapter・Interface層以外のすべてのPublicメソッド (Model, Domain Service, Value Object 等) | 局所的なロジック、不変条件、状態遷移、ビジネスルールの判断ロジック単体の検証    | `FunSpec`（シンプルなフラット構造）                  | メモリ上 (モック・外部依存なしの純粋関数)                              |
| **IntegrationTest** | 個別のAdapter（Repository、外部APIクライアント等）                                         | システム境界における1対1の接続・通信、シリアライズ、APIスキーマ等の技術的検証 | `FunSpec`                               | 実際の接続先（DB、モックサーバー等）                                 |
| **ComponentTest**   | アプリケーションサービス全体（GraphQL API等のエントリーポイント）                                      | システム単位でのブラックボックス検証、ユビキタス言語に基づくドメインシナリオの証明 | `DescribeSpec` (**Given-When-Then** 形式) | **管理下にある連携（DB等）は本物を使用**。管理外の外部連携は**モックサーバーコンテナ**を使用 |

---

## 2. UnitTest (ユニットテスト) 規約

### 特徴と方針

UnitTestは、外部依存（データベースやネットワーク等）を持たない単一クラスや純粋なドメインロジックをテストします。Given-When-Thenのような重厚なネスト構造は過剰となるため、
**フラットでシンプルな構成**を維持します。

### テストの構成とスタイル

Kotestの `FunSpec` または `StringSpec` を使用します。基本的には `FunSpec` の `test` ブロックを用いて記述します。

```kotlin
class FlightInfoTest : FunSpec({
    context("有効なフライト情報であればインスタンスが生成できること") {
        // 検証
    }
}) {
    companion object {
        val flightIdentity = FlightIdentity.create("JL123", LocalDate.of(2026, 5, 1)).shouldBeRight()
        val departurePoint = FlightPoint.create("JP", "HND", "Asia/Tokyo").shouldBeRight()
        val arrivalPoint = FlightPoint.create("US", "JFK", "America/New_York").shouldBeRight()
    }
}
```

### テストデータの定義場所

テスト内で使い回すダミーデータや共通変数は、テストのbody部分（ラムダの直下）に直接記述すると可読性が低下します。
これらはクラスの `companion object` 内に定数（`val`）として定義して、テストの見通しを良くしてください。
その際、KotestのSpecクラスはラムダコンストラクタを利用しつつ、スコープ内でコンパニオンオブジェクトを参照できるように、上記例のように
`class XTest : FunSpec({ ... }) { companion object { ... } }` の形式で定義します。これにより `init`
ブロックを省略しつつ、すっきりとした記述が可能になります。

### 対象と検証内容

**Adapter（外部依存を持つ層）以外のすべてのクラスのPublicメソッド**がUnitTestの対象となります。

* **検証内容:**
    * 有効なパラメータ（State, Command等）を与えた際に、期待する結果（新しい状態、イベント、戻り値など）が得られること。
    * 無効なパラメータを与えた際に、期待するドメインエラーが返却されること。
* **注意点:**
    * 副作用を持たない純粋関数として実装・テストするため、原則としてモックフレームワークは使用せず、メモリ上でのアサーションに特化します。

## 3. IntegrationTest (統合テスト) 規約

### 特徴と方針

IntegrationTestは、 Adapter（ControllerやRepositoryや外部サービス用のAPIクライアントなど）レイヤーなど、
管理下にあるプロセス外依存とのマッピングロジック・データクセスロジックや外部システムとの連携口になるマッピングロジックの検証を担い、
実際の接続先システムとの1対1の技術的な連携を検証する。シリアライズや通信プロトコル、スキーマ、SQLの整合性などを確認することが目的です。

### テストの構成とスタイル

Kotestの `FunSpec` を使用し、シンプルなフラット構造でアサーションを行います。
Adapter以降のロジックはモックを使用し、ビジネスロジックの検証を決して行ってはならない。

### 対象と検証内容

#### A. Outbound Adapter (Repository 等) のテスト

データベースとの接続や、SQL・O/Rマッパーの動作を検証します。

* **検証内容:**
    * ドメインモデルが正しくテーブルにマッピング・永続化され、再構築（読み込み）できること。
    * ユニーク制約違反や外部キー制約違反などのデータベース固有のエラーが正しくハンドリングされること。
    * オプティミスティックロック等の排他制御が動作すること。

#### B. External API Client のテスト

サードパーティのAPI等の外部システムとの接続やリクエスト・レスポンスのシリアライズを検証します。

* **検証内容:**
    * WireMock等のモックサーバーを使用し、想定するHTTPリクエストが送信されていること。
    * 外部レスポンス（JSON/XML等）が、システム内部のドメインモデルやDTOに正しくデシリアライズされること。
    * 接続タイムアウトや特定のエラーコード（HTTP 500等）に対するハンドリングが正しいこと。

## 4. ComponentTest (コンポーネントテスト) 規約

### 特徴と方針

ComponentTestは、システム全体を1つの「コンポーネント（単位）」とみなし、APIのエントリーポイント（GraphQL
API等）からリクエストを送信して、ブラックボックス的に全体の振る舞いを検証します。
ビジネスドキュメントとしての価値を高めるため、**Given-When-Then** 形式を徹底します。

* **管理下にあるプロセス外連携の扱い**: データベースなど、本システムが専有・管理するプロセス外連携はモックせず、*
  *本物（Docker/Testcontainers等）を使用**します。
* **管理外の外部システムの扱い (モックサーバーコンテナの活用)**:
  サードパーティAPIなど、他システムが管理する外部連携については、コード内部のインターフェース（MockK等）をモックするのではなく、
  **WireMock等のモックサーバーコンテナを立ち上げてテスト**
  します。これにより、ネットワーク通信、HTTPステータスコードの解釈、シリアライズ/デシリアライズ処理を含めた真のブラックボックス検証が可能となり、テストの網羅率と信頼性が向上します。

### テストの構成とスタイル (Kotest DescribeSpec)

Kotestの `DescribeSpec` を使用し、**Given-When-Then** 形式で構成します。

#### 基本構造

* **`describe`**: テスト対象（APIエンドポイントやビジネス上の大きなユースケース）を記述します。
* **`context("Given: ...")`**: 前提条件（データベースの初期状態、依存する外部APIのモックサーバーへのスタブ設定など）をビジネス言語で記述します。
* **`context("When: ...")`**: 実行する操作（APIへのリクエスト送信など）を記述します。
* **`it("Then: ...")`**: 期待される結果（レスポンス内容、データベースの最終状態など）を記述します。

#### 記述に関する重要なガイドライン (ブラックボックス・テストの徹底)

テストの記述（`describe`, `context`, `it`）には、**ビジネス上の関心事**のみを記述し、技術的な実装詳細を漏洩させてはなりません。

* **ユビキタス言語の使用**: 常にビジネス上の言葉（日本語を推奨）を使用し、「〜の場合」「〜したとき」といった具体的なシナリオを記述します。
* **技術用語の排除**: `GraphQL`, `MockK`, `JSON`, `HTTP 200` といったプログラミング・技術用語を記述の主たる意味として含めてはなりません（ビジネス上の結果として表現します）。

```kotlin
// ✅ ベストプラクティス例
describe("フライト監視の有効化API") {
    context("Given: すでに登録済みのフライトがある場合") {
        // DBへの事前データ登録
        // 外部フライト情報API (WireMock) が正常応答を返すスタブ設定

        context("When: 監視有効化をリクエストすると") {
            // APIクライアントによるリクエスト送信

            it("Then: 監視ステータスが有効化され、対象フライト情報が返却されること") {
                // アサーション
            }
        }
    }
}
```

## 5. Tips: データ駆動テスト (Data-Driven Testing) の方針

UnitTest (UT) および IntegrationTest (IT) において、入力値と期待値のバリエーションが多い場合は、Kotest のデータ駆動テスト（
`withData` 等）を活用します。ただし、過度な共通化による可読性の低下を防ぐため、以下のルールを遵守してください。

### 1. 適用条件（いつ使うべきか）

* **同じロジック・事前条件**に対して、入力パラメータと期待する出力だけが異なる場合に限定して使用します。（例: Value Object
  の境界値テスト、バリデーションルールの網羅など）

### 2. アンチパターン（無駄な共通化の禁止）

* **コンテキストの混在**: 「正常系」と「異常系（エラー）」、または「前提条件 (Given) が異なるケース」を無理に同じデータテーブルに押し込まないでください。
* **テストコード内での分岐**: データ側にフラグを持たせ、テストコード内で `if (data.isError)`
  のような複雑な分岐を書くのはアンチパターンです。このような場合は、正常系と異常系でデータ駆動のブロックを分けるか、別々のテストとして定義してください。

### 3. 実装のベストプラクティス（テストケースの明示的な命名）

テスト結果の可読性を高めるため、**各テストケースには必ず意味のある名前（説明）をつけてください**。Kotestの
`io.kotest.engine.names.WithDataTestName` インターフェースを実装したデータクラスを使用することで、シンプルかつ明確に実現できます（
`mapOf` を用いたマッピングよりも推奨されます）。

```kotlin
// ✅ 良い例: WithDataTestName を実装してケースに名前を持たせる
data class InvalidFlightNumberTestCase(
    val name: String,
    val invalidNumber: String
) : WithDataTestName {
    override fun dataTestName() = name
}

class FlightNumberTest : FunSpec({
    context("不正なフライト番号の場合はエラーになること") {
        withData(
            InvalidFlightNumberTestCase("文字数が短すぎる場合", "J"),
            InvalidFlightNumberTestCase("文字数が長すぎる場合", "JAL123"),
            InvalidFlightNumberTestCase("アルファベットが含まれていない場合", "12345")
        ) { testCase ->
            FlightNumber.create(testCase.invalidNumber).shouldBeLeft()
        }
    }
})
```
