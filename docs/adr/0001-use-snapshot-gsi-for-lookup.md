# ADR 0001: 関連ID検索用のルックアップ機構としてスナップショットテーブルのGSIを採用する

## Status
Accepted

## Context
イベントソーシングアーキテクチャにおいて、特定集約（例：チケット）を関連する別のID（例：UserId, FlightIdentity）で検索したいケースが存在する。
純粋なイベントストア（ジャーナルテーブル）はパーティションキーが集約ID（TicketId）であるため、そのままでは他の属性での検索ができない。

関連IDによる検索を実現するためのアーキテクチャ上の選択肢として、以下の3つが検討された。
1. **専用のルックアップテーブルの作成:** `UserId` -> `TicketId` のマッピング専用のテーブルを作成し、イベント保存時（または非同期）に更新する。
2. **ジャーナル（イベント）テーブルへのGSI追加:** イベントレコードのメタデータに `UserId` 等を持たせ、ジャーナルに直接GSIを張る。
3. **スナップショットテーブルへのGSI追加:** スナップショットテーブルに `UserId` 等のGSIを張り、そこから検索する（現在のアプローチ）。

ジャーナルテーブルへのGSI追加（案2）は、1つのチケットに紐づく複数のイベントがすべて検索にヒットしてしまい、アプリケーション側での重複排除処理が必須になること、およびGSIの書き込み・ストレージコストがイベント数に比例して膨大になることから除外された。

残る案1（専用ルックアップテーブル）と案3（スナップショットGSI）の比較となった。
案1はDDDおよびCQRSの観点で最も関心の分離が美しく理想的であるが、インフラリソース（管理テーブル数）の増加と、更新時の整合性担保（トランザクションや結果整合性）の仕組み作りが必要となる。

## Decision
現在のフェーズにおいては、既存の実装を活かし **「案3：スナップショットテーブルへのGSI追加」** をルックアップ機構として公式に採用・維持することとする。

**採用の理由・条件：**
* インフラ構成がシンプルであり、専用のルックアップテーブルを管理する手間を省けるため。
* スナップショットテーブルのレコードは集約に対して1件であるため、ジャーナルのGSIのような大量の重複結果が発生せず、検索時の処理がシンプルであるため。
* 現状の実装において、「集約の初回生成時（sequenceNumber = 1）には必ずスナップショットを作成する」というルールが組み込まれており、検索漏れ（スナップショットが存在しないために検索にヒットしない状態）が構造的に防がれているため。
* 万が一将来的に検索要件が複雑化したり、完全なCQRSへの移行が必要になった場合でも、「真の情報源（Source of Truth）」であるイベントストア（ジャーナル）さえ保全されていれば、後から過去の全イベントを読み込んで専用のルックアップテーブル（案1）やリードモデルを再構築・移行することが可能であるため。不可逆な設計上の負債にはならないと判断した。

## Consequences
**メリット:**
* テーブル構成を現状のまま維持でき、追加のインフラ構築・管理コストがかからない。
* スナップショット作成時にDynamoDBのトランザクション（`TransactWriteItems`）内で処理が完結するため、IDマッピングの同期的な整合性が容易に担保される。

**デメリット・運用上の注意点:**
* リポジトリ層での検索時（例：`findByUserId`）に、取得したスナップショットが最新状態ではない可能性がある（10回ごとのスナップショット更新を採用しているため）。
* そのため、検索結果の完全な整合性を保証するには「スナップショットのGSIで対象を特定したのち、ジャーナルから差分イベントをロードして最新状態にリプレイして返却する」という補完ロジックが不可欠である（実装・解決済み）。
* 本来は「パフォーマンス改善のためのキャッシュ」であるスナップショットテーブルに「検索のインデックス」という要件が乗り合うため、役割がやや混在していることは意識しておく必要がある。
